     1|# Vault: Standards & Lattice Design System
     2|
     3|Persistence of visual and behavioral identity for agentic operations.
     4|
     5|## 0. Foundational Axioms (Karpathy's Laws)
     6|- **Harness Over Model:** The quality of the system is defined by its evaluation arnés, not just the LLM.
     7|- **Harness Over Prompt:** Investing in environment constraints is more durable than prompt engineering.
     8|- **Trajectory as Code:** Every interaction log is data for the next generation of the agent.
     9|
    10|## 1. Lattice Design Tokens (Visual)
    11|Use these hex codes for all SVGs, diagrams, and UI components:
    12|- **Background (Deep Sea):** `#0f172a`
    13|- **Action (Neon Cyan):** `#22d3ee`
    14|- **Knowledge (Emerald Gate):** `#34d399`
    15|- **Strategy (Mystic Violet):** `#a78bfa`
    16|- **Alert (Amber Warning):** `#f59e0b`
    17|- **Error (Crimson Failure):** `#ef4444`
    18|
    19|## 2. Agent UX Components
    20|When the agent generates an interface or report:
    21|- **Header:** Title in JetBrains Mono, Bold.
    22|- **Charts:** Use Dark Mode by default.
    23|- **Tone:** William of Baskerville (Analytical/Sober).
    24|
    25|## 3. Naming Conventions
    26|- **Files:** `kebab-case` (e.g., `monastic-schedule.md`).
    27|- **Commits:** Conventional Commits (`feat:`, `fix:`, `docs:`, `chore:`).
    28|- **Tasks:** `[ID] Action: Target` (e.g., `[T-001] Refactor: MCP API`).

### Harness Engineering (Karpathy/Osmani Standard)
- **Harness Over Model:** The quality of the system is limited by its harness, not just the model.
- **Software 3.0 Principles:** Transitioning from writing code to curating agentic loops and datasets.
